# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

MPV Media Engine (package name `mvp_sound_engine`) is a Flutter Windows desktop app that acts as a real-time control surface for the [MPV media player](https://mpv.io/). It never edits MPV's config files — instead it pushes property/filter changes live over MPV's JSON IPC, so shaders, HDR tone mapping, scalers, and FFmpeg audio DSP chains (EQ, compressor, DynAudNorm, pan/downmix, ambience) all update instantly while a video is already playing.

### Critical rules for running/building (do not violate)
1. **Never run `flutter run` yourself** (directly or via a background tool) for this Windows desktop app — the GUI launches invisibly in the background session and you can't interact with or verify it. Always ask the user to run it themselves, or tell them to double-click `run_app.bat`.
2. **Always ensure zombie processes are killed before building or running.** `MPV_Media_Engine.exe` (the `BINARY_NAME` in [windows/CMakeLists.txt](windows/CMakeLists.txt)), `dart.exe`, and `dartaotruntime.exe` frequently stay alive and lock the executable, causing `LNK1168` link errors. `run_app.bat` already does this cleanup (`taskkill /F /IM ...`) before invoking `flutter run`.
3. **If you need to verify the app builds, use `flutter build windows` only** — do not run the resulting executable yourself.

## Architecture

### Provider / state layering
The app uses `provider` (`ChangeNotifier`) for state, wired up in [lib/app.dart](lib/app.dart):
- `DspProvider` — owns the audio DSP state (`DspState` in [lib/models/dsp_state.dart](lib/models/dsp_state.dart)) and the single `MpvIpcService` connection. It's the only provider that actually owns the IPC socket.
- `VideoProvider(DspProvider)` — owns video/shader/HDR state (`VideoState` in [lib/models/video_state.dart](lib/models/video_state.dart)) but has **no IPC connection of its own** — it sends every command through `dspProvider.sendRawCommand(...)`, so `DspProvider` must be constructed first.
- `ThemeProvider` — selects one of three `AppThemeMode`s (`dark` default, `teal`, `light`), persisted via `PreferencesService`. Note the theme is *also* mirrored into a static global, `AppTheme.mode`, which every colour getter in [lib/constants/theme.dart](lib/constants/theme.dart) switches on — so widgets read `AppTheme.background` etc. directly rather than going through `Theme.of(context)`. `app.dart` reassigns `AppTheme.mode` on each rebuild; anything reading colours outside a `Consumer<ThemeProvider>` subtree won't repaint on theme change.

### Screen / tab structure
Video is the app's headline feature and owns the main tab bar in [lib/screens/home_screen.dart](lib/screens/home_screen.dart): **Shaders, HDR & Tone Mapping, Scaling & Interpolation, Grading & Deband, Debug IPC** (`tab_video_*.dart` + `tab_debug.dart`). Audio is *not* a tab — `SoundSettingsEntry` (a card next to the Video Preset selector) pushes [lib/screens/sound_settings_screen.dart](lib/screens/sound_settings_screen.dart) full-screen, which wraps `TabSound` and its own five sub-tabs (Loudness & Dynamics, Channels & Stereo, Ambience & Space, EQ & Tone, Safety). The Video Preset chrome is hidden on Debug IPC via `_videoChromeHiddenOnTabs`. `video_controls_common.dart` holds the shared slider/dropdown row widgets used across the four video tabs.

### DSP audio pipeline (DspProvider)
- Every setter (`setEqBandGain`, `setCompThreshold`, `setBypass`, etc.) mutates `DspState` immutably via `copyWith`, rebuilds a preview string, and — if `autoApply` is on — debounces (120ms) a call to `_applyNow()`.
- [lib/services/filter_builder.dart](lib/services/filter_builder.dart) (`FilterBuilder`) turns `DspState` into a single FFmpeg `lavfi` filter-chain string (`dynaudnorm,pan,asplit/aecho ambience,extrastereo,anequalizer,highshelf,acompressor,alimiter`) and into the actual MPV IPC JSON command (`set_property af <chain>`).
- [lib/services/filter_parser.dart](lib/services/filter_parser.dart) (`FilterParser`) does the reverse: parses a raw filter string back into a `DspState` so the GUI sliders stay in sync when a custom/pasted filter string is loaded.
- Presets (`Preset` in [lib/models/preset.dart](lib/models/preset.dart)) bundle a `DspState` + optional raw `customFilter` override; user-saved ones persist through `PreferencesService`/`shared_preferences` as JSON. Video has a parallel `VideoPreset` ([lib/models/video_preset.dart](lib/models/video_preset.dart), built-ins in `builtinVideoPresets`) stored under its own prefs key.

### Video pipeline (VideoProvider)
- Unlike DSP, video properties are mostly set directly as individual MPV `set_property` commands (tone-mapping, target-peak, brightness/contrast/gamma, deband, tscale/scale/cscale/dscale, glsl-shaders), not built into one filter string.
- `applyPreset()` diffs the outgoing preset's `VideoState` against the currently-live one and only enqueues properties that actually changed — resending unchanged scale/shader/interpolation values forces needless libplacebo pipeline rebuilds. The **first** `applyPreset()` call after each new MPV (re)connection forces a full unconditional send regardless of the diff (`_needsFullResync`, reset via a `dspProvider` connection-state listener) — local `VideoState` is only ever a *guess* at MPV's live properties (a freshly connected MPV instance may already have shaders/scalers set from its own `mpv.conf` or a prior session), so the diff can't be trusted until we've done at least one confirmed full sync.
- Shader paths are resolved to absolute paths at `assets/shaders/` (or `data/flutter_assets/assets/shaders/` in release builds) before being sent, since MPV needs real filesystem paths, not Flutter asset keys.
- **Shader chains are per resolution tier and mutually exclusive at runtime.** `VideoState` holds `shadersLowRes` (≤1080p) and `shadersHighRes` (1440p+; boundary `dheight >= 1300` via `getResolutionTier`), and only the list matching the *current video's* tier is ever sent as `glsl-shaders` — the Shaders tab badges the other tier's section "Not applied" but **keeps it editable** (`setShaders` only updates stored state for the non-live tier). Fully disabling the inactive section shipped briefly and backfired: while a 4K video played you couldn't clear a stale ≤1080p list, so it silently sprang back to life on the next 1080p video. Consequences: `_buildStateCommands` diffs only the live tier's list; `cacheCurrentVideoInfo` re-sends the chain when a new video crosses the tier boundary; new-file detection lives in `VideoProvider._maybeRefreshVideoInfo`, **not** the tab widget (`TabBarView` disposes hidden tabs, so detection there died whenever another tab was open); `VideoState.fromJson` migrates the pre-split flat `activeShaders` list by each shader's `recommendedFor` metadata. The two lists share the one mpv property, which `video_state_coverage_test.dart` accounts for (a field is covered if it emits under *either* tier; the forceAll test expects fields−1 properties). `shader_tier_test.dart` pins the gating and the migration.
- On startup, `_checkWindowsHdr()` calls the native platform channel to auto-default to full HDR passthrough if Windows HDR is currently enabled.
- **HDR Output** (`setHdrOutput`) is a GUI-only concept with no matching mpv property — it's a "force full HDR passthrough" shortcut that drives real properties underneath: `target-colorspace-hint=yes` + `target-trc=pq` + `inverse-tone-mapping=yes` when on, `target-trc=auto` + `inverse-tone-mapping=no` when off. It also gates two other controls: the **Visualizer** switch (`tone-mapping-visualize` draws nothing unless HDR Output is on, so it's disabled/force-off otherwise) and the separate **Target Hinting** switch (disabled while HDR Output is on, since HDR Output already owns `target-colorspace-hint`/`target-trc` and the two would otherwise fight over the same mpv state). Turning HDR Output **off releases everything it owns** (`hint=no`, `trc=auto`, `itm=no`, `target-peak=203`; on-state forces `hint=yes`/`trc=pq`/`itm=yes`/`target-peak=auto`) — it used to leave the hint stuck at the `yes` that enabling wrote, stranding mpv in absolute-PQ output where **Target Peak visibly does nothing** for values above the content's own peak (SDR content pins at ~203 nits; without inverse-tone-mapping mpv never expands upward — confirmed by reading `video-target-params` back: gamma stayed `pq` under the stale hint). Windows-HDR machines auto-enable HDR Output at startup, so one manual toggle-off used to strand every such user there; `video_state_coverage_test.dart` pins the release.

### MPV IPC transport
- **All outbound IPC commands funnel through one FIFO queue in `DspProvider`** (`sendRawCommand` → `_outbox` → `_drainOutbox()`), regardless of which provider/widget originated them. This is deliberate: MPV/libplacebo can freeze if hit with an unpaced burst of property changes, and having every call site independently manage its own `Timer`-based delay (the old approach) meant overlapping actions (e.g. switching presets while a slider drag was still debouncing) could interleave into a burst anyway. The queue enforces a minimum gap between *every* pair of consecutive sends (`_kMinCommandGap`, 150ms), with an optional longer `minGapAfter` for properties known to trigger a full render-pipeline rebuild (`scale`/`cscale`/`dscale`/`tscale`/`glsl-shaders`/`interpolation`/`video-sync` — see `VideoProvider._kExpensiveProperties`).
- `MpvIpcService` (Desktop): connects via raw TCP socket (default `127.0.0.1:9001`) or a `WebSocketChannel` (default `ws://127.0.0.1:9002`); Web (`kIsWeb`) always uses WebSocket. `dart:io` is behind a conditional import (`lib/stubs/io_stub.dart` stands in for `dart:io` on web builds), so any `dart:io` usage in shared code must go through the `io.` prefix import pattern already used in `mpv_ipc_service.dart`/`dsp_provider.dart`/`video_provider.dart`.
- **⚠️ mpv's `--input-ipc-server` on Windows is always a named pipe, never a raw TCP socket** — confirmed directly against mpv's own manual and by empirically launching mpv with `tcp://…` (it fails to bind, since mpv literally treats an unrecognized scheme as a literal pipe-name suffix). `MpvIpcService.connect()`'s `127.0.0.1:9001` TCP candidate and the "direct TCP" path only work if something *else* is proxying TCP↔pipe — they cannot connect to a real `mpv.exe` on Windows by themselves. The WebSocket bridge scripts exist for exactly this reason.
- **[mpv_websocket_bridge.ps1](mpv_websocket_bridge.ps1) proxies MPV's named pipe ↔ a WebSocket** the app can actually connect to. `DspProvider.playTestVideo()` launches mpv with `--input-ipc-server=\\.\pipe\<random>`, unpacks the bridge to the system temp dir (rewriting the `$wsPort`/`$pipeName` literals inside the script text) with a per-session random WebSocket port and pipe name to avoid collisions, retries the WS connection up to 10 times, then applies the current DSP state once connected. **Nothing needs to live next to `mpv.exe`** — the bridge always runs from temp.
- **PowerShell is the only bridge, deliberately — do not add a Python (or other interpreter) fallback back in.** A Python bridge shipped through v1.3.0 and was removed because it could never work: it spoke TCP to `127.0.0.1:9001`, which per the point above no real Windows mpv ever listens on. It also *masked* its own failure — on a machine with no real Python, `python` silently resolves to the Windows Store app-execution-alias stub, which spawns without throwing and so looks like a successful bridge start. That combination was the v1.3.0→v1.3.1 connect-failure bug. `powershell.exe` is present on every Windows install, so a fallback buys nothing.
- **The bridge is bidirectional**: it keeps one read pending on mpv's pipe at all times and forwards every reply/event line back over the WebSocket, so `get_property` round-trips work (`MpvIpcService.getProperty` correlates replies via `request_id`). **⚠️ The named pipe must be opened with `PipeOptions.Asynchronous`** — without it the handle is non-overlapped and Windows serializes operations on it, so one parked read blocks *every* write and all IPC commands (volume, EQ, presets) freeze until mpv happens to emit a line. This exact bug shipped briefly and broke the whole app; don't remove that flag. The bridge also logs to `%TEMP%\mpv_bridge_<pipeName>.log`, which matters because `DspProvider.playTestVideo()` drains and discards the bridge's stdout. Test any bridge change in isolation first (bare `mpv --idle` + a copy of the bridge on a throwaway pipe/port + a standalone WebSocket client) before wiring it into the app.

### Command rejection reporting
- **Every outbound command is stamped with a unique `request_id`.** `MpvIpcService.sendCommand()` injects one into any payload that doesn't already carry it (`getProperty` supplies its own), and records a description of the command in `_inFlightCommands`. mpv answers *every* command with an `error` field, so when a reply comes back non-`success`, the exact command that caused it can be named. Both fire-and-forget commands and `getProperty` draw from the same `_requestIdCounter`, so ids are globally unique and a command's ack can never wrongly complete a pending property query.
- Rejections are emitted on `MpvIpcService.commandErrorStream` → `DspProvider` logs them as `❌ mpv REJECTED: …` and keeps the last 20 in `commandErrors`, which the Debug IPC tab renders as a red "Rejected by mpv (N)" panel (hidden when empty).
- **Why this exists:** a rejected `set_property` is otherwise invisible — mpv returns an error, nothing throws, and the control just silently does nothing. `hdr-output` (a property that doesn't exist) was a dead toggle for months because of this. The Debug IPC tab has a permanent **Test Error Surfacing** button that sends a deliberately bogus property; if pressing it does *not* light up the red panel, rejection reporting itself is broken and this whole class of bug is invisible again.

### Startup flow (first run, mpv path, updates)
- The app needs to know where `mpv.exe` is; `DspProvider.needsFirstTimeSetup` (true once prefs load and no path is stored) triggers `showFirstRunSetupIfNeeded` ([lib/widgets/first_run_setup.dart](lib/widgets/first_run_setup.dart)). That's the *only* thing first-run setup collects — `setMpvExePath()` just persists the path.
- The dialog pre-fills that path from [lib/services/mpv_locator.dart](lib/services/mpv_locator.dart), which searches cheapest-first and returns on the first hit: (1) ~14 exact `existsSync` probes of conventional install dirs; (2) `where.exe` for PATH, which is where scoop/chocolatey/winget put their shims; (3) `reg.exe` against `Applications\mpv.exe\shell\open\command`, `App Paths` and **MuiCache** — Windows records the full path of every executable it has seen *run*, which is the only cheap way to find a portable build in an arbitrary folder; (4) a **depth-1** sweep of a few likely parents on every drive (`fsutil fsinfo drives`), matching any child folder named `mpv*`, which catches an mpv that was extracted somewhere odd and *never launched* (so steps 1–3 can't know about it) — e.g. shinchiro's archive unpacks to `mpv-x86_64-v3-<date>-git-<sha>\`.
  - **It must never walk the filesystem, and it must stay fast** (`test/mpv_locator_test.dart` guards both). The whole search is capped at 1.5s, each helper process at 600ms. Step 4 is fully **async** and scans drives **concurrently** on purpose: a `listSync`/`existsSync` against a dead mapped network drive blocks for the SMB timeout and would wedge the isolate somewhere no `.timeout()` could reach, and a sleeping mechanical drive can take ~700ms to spin up — concurrency stops one slow drive starving the others of the budget.
  - **A null result means "couldn't find it", never "you don't have mpv"** — the search can't prove absence. The not-found panel therefore leads with **Browse** and offers the mpv.io download only as a secondary action; telling a user with a portable mpv to go download a second copy would be wrong.
- [lib/services/update_checker.dart](lib/services/update_checker.dart) polls the GitHub Releases API for `ddinh99/MPV-Media-Engine` against the `package_info_plus` version and surfaces a dismissible banner (dismissal persisted per-version). It must never throw or block startup — it swallows every error and returns null.

### Windows native layer
- [windows/runner/flutter_window.cpp](windows/runner/flutter_window.cpp) registers a `MethodChannel` named `com.mpv_media_engine/platform` that Dart calls through [lib/services/platform_service.dart](lib/services/platform_service.dart). It exposes `isWindowsHdrEnabled` (via DXGI `IDXGIOutput6::GetDesc1`, checking for `DXGI_COLOR_SPACE_RGB_FULL_G2084_NONE_P2020`) and `getDisplayMaxLuminance` (same `GetDesc1`, returning `MaxLuminance` in nits — preferring an HDR-mode output — which the HDR tab draws as a display-peak marker on the Target Peak slider; 0 means unknown and maps to null in Dart). Linking against `dxgi.lib` is declared in [windows/runner/CMakeLists.txt](windows/runner/CMakeLists.txt).
- Any new native platform capability should be added as another `call.method_name()` branch in the same channel handler rather than creating a new channel.

## Verified mpv property gotchas
Before adding/changing any mpv property name or value, cross-check it against a real mpv binary's `--list-options` (`mpv.exe --list-options`) rather than trusting memory or old snippets — several were wrong and shipped silently broken. A rejected `set_property` doesn't crash anything, so historically a wrong property name just looked like "the slider does nothing". **Rejections are now surfaced** (see *Command rejection reporting* below), but that only helps if you actually look: check the Debug IPC tab's red "Rejected by mpv" panel after touching any property.
- `hdr-output` is **not a real mpv property** (confirmed: mpv errors "option not found" even at the CLI). There is no such thing — HDR passthrough is expressed via `target-colorspace-hint` + `target-trc` + `inverse-tone-mapping` instead (see `VideoProvider.setHdrOutput`).
- Contrast Recovery is `hdr-contrast-recovery` (range 0–2) — **not** `tone-mapping-contrast-recovery` or `contrast-recovery`, neither of which exist.
- `--tone-mapping` has no `none` value (`Invalid value` error); the real "disable" value is `clip`. The GUI keeps "None" as a friendly label but translates it via `_mpvToneMappingValue()` before sending.
- `tscale-window`'s value is `hanning`, not `hann`.
- **`target-peak` is an integer option and mpv's JSON IPC will not coerce a double** — `set_property target-peak 203.0` returns "error accessing property" while `203` succeeds (verified against mpv 0.41). Dart doubles serialize as `203.0`, so sending `VideoState.targetPeak` raw made the Target Peak slider (and every preset's targetPeak) silently dead; both send sites now `round()` first, and `video_state_coverage_test.dart` pins the wire type. Value *types* matter over IPC, not just names — the coverage test only proves a command is emitted, not that mpv accepts it. The other double-typed fields are safe: `hdr-contrast-recovery`, `hdr-peak-percentile`, `tscale-radius/blur/clamp` are declared Float.
- **An explicit `target-peak` under HDR passthrough is an absolute PQ output ceiling** — with `target-colorspace-hint=yes` + `target-trc=pq`, `video-target-params.max-luma` follows `target-peak` exactly, so the 203 the passthrough presets used to carry crushed HDR to 203 nits on a 1450-nit panel the moment the dead slider was fixed (until then it was silently rejected, and everyone unknowingly ran mpv's `auto`). `VideoState.targetPeak == 0.0` is the **auto sentinel** (wire value: the string `'auto'`, runtime-settable — verified) — passthrough presets and `setHdrOutput(true)` must use it; SDR presets keep explicit 203 (≈ what auto resolves to on the SDR path). `VideoState.fromJson` migrates the poisoned saved combo (`hdrOutput: true` + `targetPeak: 203`) to auto, since that pairing predates the slider working. On the SDR path the slider is *inverted* relative to intuition (it describes the display: higher claimed peak → dimmer picture); under HDR Output it's an expansion target (higher → brighter). **With `tone-mapping=clip` (the GUI's "None", used by the passthrough presets) the slider is inert on SDR content** — measured bit-identical output at every value — and only trims highlights on HDR content; there's no remapping curve for target-peak to reshape. The HDR tab shows a caption for this state instead of leaving a dead-feeling control.
- **`hdr-reference-white` has the exact same integer-or-`'auto'` wire type as `target-peak`, and the same double-rejection trap** — verified directly against a live mpv 0.41 IPC connection (not just `--list-options`): `set_property hdr-reference-white 150.0` returns `"error accessing property"` while `150` succeeds. `VideoState.hdrReferenceWhite == 0.0` is the auto sentinel (wire value `'auto'`); `VideoProvider._mpvHdrReferenceWhiteValue()` rounds before sending, same as `_mpvTargetPeakValue()`.
- `--tone-mapping` only actually inserts a mapping filter when the *loaded content's own* peak/gamut exceeds the configured `target-peak` (i.e. it's content-dependent, not something any GUI toggle alone determines) — see mpv's own manual wording under `--tone-mapping` and `--target-peak`.
- `--inverse-tone-mapping` must be explicitly enabled for mpv to expand dynamic range at all (SDR→HDR, or brightening HDR further); without it, the Algorithm dropdown does nothing in that direction regardless of which curve is selected.
- **`video-sync=display-resample` will race the video ahead of the audio if mpv can't measure the display.** Interpolation requires a `display-*` sync mode, and `display-resample` retimes video against **`estimated-display-fps`** — the refresh mpv *measures*, not the nominal `display-fps`. When that measurement runs away (observed at **347** and **2817** against a real 60Hz panel), mpv faithfully sprints the video to chase it: audio stays at normal speed, video runs several times too fast, and `avsync` hit **−14s within 4s of playback**.
  - **The variable is whether the GPU driver actually vsyncs, not what mpv asks for.** Confirmed on real hardware: driver V-Sync on → mpv measures the display correctly → interpolation works. Driver V-Sync off → the measurement runs away → video races. Beware that `--d3d11-sync-interval=1` is only mpv *requesting* vsync; a driver-level "V-Sync: Off" overrides it, so that flag is **not** a way to test the vsync-on case (mistaking it for one produced a bogus "reproduces with vsync on too" conclusion here). Per-application driver profiles are keyed to the *executable path*, so pointing the app at a different mpv.exe silently loses the profile — which is exactly how this surfaced.
  - mpv **accepts** `display-resample` without error and only *then* misbehaves, so this emits no rejection and the Debug IPC red panel stays silent — same invisible-failure class as the dead `hdr-output` toggle. The only defence is to stop trusting the write: `VideoProvider._verifyDisplaySync()` reads `display-fps`/`estimated-display-fps` back 3s after interpolation is enabled and, if the measurement isn't credible (>1.5× nominal), reverts to `video-sync=audio` + `interpolation=no` and surfaces a warning. `video-sync=audio` played correctly in *every* configuration tested, including vsync off.
  - **Every path that can turn interpolation on must run the check**, not just the toggle. `_resyncAll()` pushes the whole *persisted* state to a freshly connected mpv, so a saved `interpolation: true` reaches a binary we have never measured without anyone calling `setInterpolation()`/`applyPreset()` — that hole let the video race indefinitely rather than for one settle window. The verifier also retries (`_kDisplaySyncAttempts`), because a resync fires on *connect*, which can land before playback starts and before mpv has measured anything.
  - **Changing the mpv.exe path resets interpolation** (`_checkMpvBinaryChanged`). Driver settings are keyed to the executable path, so a V-Sync override for one mpv.exe does not follow you to another — display sync working on the old binary says nothing about the new one. Note the first sighting of a path is prefs loading at startup, *not* a user switch; treating it as one would reset interpolation on every launch.
  - Wall-clock duration is **blind** to this: audio still gates the end of file, so the video racing ahead doesn't change how long playback takes. Measure `avsync`/`time-pos` over IPC, not elapsed time — several hours were lost to "correct" 7.1s timings while the video was sprinting.

## Notes on tests
Most of `test/` is minimal/exploratory — **don't assume `flutter test` output reflects real regressions without checking**. The exceptions:
- **`display_sync_guard_test.dart` is real and load-bearing — keep it green.** It pins the fallback that stops `video-sync=display-resample` racing the video ahead of the audio (see the `display-resample` entry under *Verified mpv property gotchas*). It fakes mpv's `estimated-display-fps` answers and asserts that a runaway measurement forces interpolation back off *and* that the fallback actually reaches mpv — plus that a genuine 144Hz panel isn't mistaken for a runaway.
- **`mpv_locator_test.dart` guards a speed property, not a result.** It asserts the mpv search finishes well inside its budget and never throws, because it runs while the first-run dialog is on screen. It deliberately does *not* assert that mpv is found — that depends on the machine. If it starts failing, someone probably made the search recursive.
- **`video_state_coverage_test.dart` is real and load-bearing — keep it green.** It guards the bug class that silently broke HDR Output: a `VideoState` field the GUI reads/writes that never actually reaches mpv (no `addIfChanged` in `_buildStateCommands`, so the control does nothing), or that isn't in `toJson`/`fromJson` (so it resets on every launch). It walks `toJson()`'s keys as the field inventory — Flutter has no runtime reflection — mutates one field at a time, and asserts a command is emitted. **If you add a field to `VideoState`, this test tells you what you forgot to wire.** A field with genuinely no mpv property (e.g. `hdrOutput`) goes in the test's `_guiOnlyFields` map *with a reason*; don't silently widen that allowlist to make a failure go away. Its blind spot: a field missing from `toJson` entirely is invisible to it.
- `widget_test.dart` asserts the header text is `'MVP Sound Engine'`, which no longer matches the actual header (`'MPV Media Engine'`). This test fails today, and has nothing to do with whatever you just changed.
- `preferences_test.dart` only mocks `SharedPreferences`; it asserts nothing meaningful.
- `test_ws.dart` and `revert_helper.dart` are not tests at all — they're throwaway `main()` scripts (the latter hardcodes an absolute path to a long-gone IDE transcript). `flutter test` skips them since they don't end in `_test.dart`.

## Naming drift
The product is **MPV Media Engine** and the built binary is `MPV_Media_Engine.exe`, but the Dart package is still `mvp_sound_engine` (note *MVP*, not *MPV*) — imports must use `package:mvp_sound_engine/...`, and the root class is `MvpSoundEngineApp`. The repo dir and release zips say `MPV-Sound-Engine`; the GitHub repo is `MPV-Media-Engine`. Don't "correct" the package name to match the product — the rename would touch every import.
